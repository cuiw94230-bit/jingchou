# 推演功能后端接口文档

最后更新：2026-04-23

## 1. 说明

本文档记录当前项目中“推演”相关后端能力的真实落地状态。  
后续只要推演逻辑、库表、接口字段、日志链路发生变化，都必须同步更新本文档。

当前推演已拆成两个模式：

1. 单店时间推演  
只改变 1 家店在目标波次中的期望到店时间，分别跑一遍“改前基线求解”和“改后模拟求解”，做 A/B 对照。

2. 减车可行性推演  
以目标车辆数和选定波次为约束，直接跑真实求解，输出失败样本、约束诊断和规则建议。

这两个模式共用 `/simulate/optimize-time` 入口，但业务语义已不同，后端必须按 `mode` 区分处理。

## 2. 当前已落地接口

### 2.1 GET /simulate/config

功能：
- 获取推演配置
- 返回波次规则、允许晚到上限、时间步长等基础配置

当前返回为默认配置。

响应示例：

```json
{
  "success": true,
  "data": {
    "max_late_minutes": 60,
    "time_step_minutes": 30,
    "default_target": "min_vehicles",
    "wave_adjust_enable": true,
    "wave_rules": {
      "W1": {"start": "19:10", "end": "23:59", "allow_cross_wave": true},
      "W2": {"start": "21:00", "end": "07:00", "allow_cross_wave": true},
      "W3": {"start": "21:00", "end": "07:00", "allow_cross_wave": false},
      "W4": {"start": "12:00", "end": "18:00", "allow_cross_wave": false}
    }
  }
}
```

### 2.2 POST /simulate/optimize-time

功能：
- 推演统一入口
- 当前支持两种模式：
  - `single_store`：单店时间推演
  - `fleet_feasibility`：减车可行性推演

#### 2.2.1 单店时间推演

业务语义：
- 读取当前真实求解数据通道，而不是单纯吃历史展示结果
- 对目标店只修改 1 个变量：目标波次下的期望到店时间
- 用同一批输入各跑两次：
  - 基线求解：改前时间
  - 模拟求解：改后时间
- 输出 A/B 对照、路线变化、调度到店时间来源说明、失败样本和过程日志

请求示例：

```json
{
  "batch_id": "run-1776672792002",
  "mode": "single_store",
  "adjustments": [
    {
      "shop_code": "391001",
      "old_wave": "W1",
      "old_time": "21:30",
      "new_wave": "W1",
      "new_time": "22:30",
      "new_allowed_late": 30
    }
  ],
  "target": "min_vehicles",
  "include_route_details": true
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `batch_id` | string | 是 | 源批次 ID |
| `mode` | string | 否 | 默认为 `single_store` |
| `adjustments` | array | 是 | 门店调整列表 |
| `adjustments[].shop_code` | string | 是 | 店铺编号 |
| `adjustments[].old_wave` | string | 否 | 原波次，仅用于校验与展示 |
| `adjustments[].old_time` | string | 否 | 原时间，仅用于校验与展示 |
| `adjustments[].new_wave` | string | 是 | 模拟波次 |
| `adjustments[].new_time` | string | 是 | 模拟时间，格式 `HH:MM` |
| `adjustments[].new_allowed_late` | int | 否 | 新允许晚到分钟数 |
| `target` | string | 否 | `min_vehicles` / `balanced` |
| `include_route_details` | bool | 否 | 是否返回按波次汇总的线路对比详情 |

响应主体当前包含：

```json
{
  "success": true,
  "task_id": "sim_1776929501688_dw63kg",
  "data": {
    "mode": "single_store",
    "selected_waves": ["W1"],
    "before": {},
    "after": {},
    "improvements": {},
    "route_comparison": {},
    "schedule_diagnostics": {},
    "ab_compare": {},
    "audit_proof": {},
    "report": {},
    "route_changes": {},
    "attempt_trace_summary": {},
    "failure_logs": []
  }
}
```

其中重点字段说明：

- `before` / `after`
  - A/B 两次求解后的汇总指标
  - 包含总车次、总里程、各波次统计

- `route_comparison`
  - 目标店相关线路对比
  - 包含旧线路、改后线路、逐店顺序、期望到店时间、调度到店时间、货量等

- `schedule_diagnostics`
  - 调度到店时间来源说明
  - 标识是：
    - `solver_returned`
    - `post_processed`
    - `missing`

- `ab_compare`
  - 单波次 A/B 对照结果
  - 不是字段名回显，而是基线求解和模拟求解在候选门店、已安排门店、未安排门店、车辆、里程上的对比

- `audit_proof`
  - 证明 A/B 两次求解的唯一变量是目标店时间变化

- `report`
  - 中文报告摘要
  - 供前端弹窗和“推演结果”模块直接使用

- `attempt_trace_summary`
  - 当前任务中试插回放记录数量

- `failure_logs`
  - 当前模式下若出现失败样本，返回失败约束、中文建议、依据说明

#### 2.2.2 减车可行性推演

业务语义：
- 目标是验证“指定车辆数 + 指定波次”能否完成求解
- 当前走真实求解
- 会输出失败样本、试插过程、中文约束诊断和建议

请求示例：

```json
{
  "batch_id": "run-1776672792002",
  "mode": "fleet_feasibility",
  "target_vehicle_count": 8,
  "selected_waves": ["W1", "W2"],
  "target": "min_vehicles",
  "include_route_details": true
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `batch_id` | string | 是 | 源批次 ID |
| `mode` | string | 是 | 固定为 `fleet_feasibility` |
| `target_vehicle_count` | int | 是 | 目标车辆数 |
| `selected_waves` | array | 否 | 指定波次，默认按后端策略 |
| `target` | string | 否 | `min_vehicles` / `balanced` |
| `include_route_details` | bool | 否 | 是否返回按波次汇总详情 |

响应重点字段：

- `before` / `after`
- `failure_logs`
- `attempt_trace_summary`
- `route_changes`

说明：
- 这里的 `vehicles` 是当前统计口径下的“车次/排线结果统计”
- 后续若前端需要明确展示“独立车号数量”与“车次数量”两种口径，需单独补字段，不要混用

### 2.3 GET /simulate/task-log

功能：
- 获取某个推演任务的后台过程日志
- 前端通过轮询该接口显示“求解过程可视化窗口”

请求参数：

| 参数 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `taskId` | string | 是 | 推演任务 ID |
| `cursor` | int | 否 | 增量游标 |

返回说明：
- 返回日志行数组和新的游标
- 用于前端增量追加日志，不重复拉取

## 3. 当前真实数据通道

### 3.1 单店时间推演

当前数据来源要点：

- 门店：优先读取当前真实求解调用表
- 车辆：依赖操作者在真实调度前导入的车辆列表
- 波次与求解参数：从实时上下文和源批次中组合
- 调度算法：复用真实求解链路，而不是前端假排

重要约束：

- 操作者必须先在调度页完成当前车辆导入、货量保存等动作
- 然后再打开推演页，否则推演用到的实时车辆/门店数据不完整

### 3.2 减车可行性推演

当前数据来源要点：

- 目标车数来自用户输入
- 剩余门店、车辆、距离矩阵都应来自当前真实求解上下文
- 失败样本与试插回放来自本次真实推演过程，而不是历史失败记录

## 6. 中文诊断与建议口径

失败样本展示必须面向业务人员，不能只显示字段名。

推荐口径：

### 6.1 时间窗超时

```text
W1 | 店391001 | 时间窗超时 18 分钟 | 当前到达 22:48 > 最晚允许 22:30
建议A：门店时间调到 22:48
建议B：至少提前 18 分钟
建议C：尝试改派相邻波次（若允许跨波次）
依据：失败类型为时间窗超时，触发规则为到店时间窗检查，关联车辆 京A12345
```

### 6.2 波次结束超时

```text
W1 | 店392712 | 波次结束超时 39 分钟 | 当前结束 00:38 > 截止 23:59
建议A：将 W1 截止从 23:59 放宽到 00:38（+39分钟）
建议B：店铺提前到 22:51（-39分钟）
建议C：尝试改派相邻波次（若允许跨波次）
依据：失败类型为波次结束超时，触发规则为波次结束时刻检查，关联车辆 京N7MR82
```

注意：
- 跨天时间必须正确显示 `D+1`
- 不能出现“01:38 明明早于 07:00 却报超时”的跨天错账

### 6.3 里程超限

```text
W2 | 店398439 | 里程超限 | 当前线路里程 149.6 km，限制 140.0 km，超出 9.6 km
建议A：将该线路里程上限放宽到 149.6 km
建议B：优先挪出相邻远端门店，缩短该线
建议C：尝试改派相邻波次或其他车辆
依据：失败类型为里程超限，触发规则为单程里程检查，关联车辆 京N0NS61
```

说明：
- 里程建议必须给出“建议放宽到多少公里”
- 不能只写“缺口 0.0 km”这类业务不可决策的话

### 6.4 装载超限

```text
W1 | 店394089 | 装载超限 | 当前装载 1.26，限制 1.20，超出 0.06
建议A：将装载上限从 1.20 放宽到 1.26
建议B：拆分本线路中高货量门店
建议C：调拨到剩余车辆或相邻波次
依据：失败类型为装载超限，触发规则为车辆容量检查，关联车辆 京Q55WU6
```

## 7. 当前前端展示口径要求

### 7.1 单店时间推演

必须展示：

1. A/B 中文报告
2. 改前线路详情
3. 改后线路详情
4. 每个停靠点的：
   - 店铺编号
   - 店名
   - 期望到店
   - 调度到店
   - 货量
5. 调度时间来源说明

关键要求：
- 改后线路中目标店的期望到店时间必须显示为推演时间
- 如果求解器没返回时间，要明确标注来源，不得伪装成真实求解时间

### 7.2 减车可行性推演

必须展示：

1. 总体结论  
例如：  
“我们尝试用 8 辆车满足 W1/W2 任务，推演结果为：W1 有 8 家店无法安排，W2 有 3 家店无法安排。”

2. 规则建议  
例如：
- 哪些店建议改到几点
- 哪条规则建议放宽到多少公里
- 哪个波次截止建议放宽到几点

3. 过程日志可视化

4. 失败样本与试插回放

## 8. 错误码

| 错误码 | 说明 |
| --- | --- |
| `INVALID_BATCH_ID` | 批次 ID 不存在 |
| `INVALID_SHOP_CODE` | 店铺编号不存在 |
| `INVALID_TIME` | 时间格式错误或超出波次范围 |
| `INVALID_WAVE` | 波次不存在或不可跨波次 |
| `INVALID_VEHICLE_COUNT` | 目标车辆数非法 |
| `STORES_EMPTY` | 当前求解调用表无门店数据 |
| `VEHICLES_EMPTY` | 当前未导入有效车辆 |
| `RESCHEDULE_FAILED` | 重新调度失败 |
| `AB_COMPARE_EMPTY` | 未生成有效 A/B 对照结果 |
| `TASK_NOT_FOUND` | 任务 ID 不存在 |
| `TASK_FAILED` | 推演任务执行失败 |

## 10. 2026-04-23 补充

### 10.1 新增接口：GET /simulate/single-report

用途：

- 单店时间推演完成后，前端报告页和“推演结果”弹窗不再直接依赖 `/simulate/optimize-time` 当次返回。
- 改为按 `taskId` 从数据库读取已经落库的单店推演审计结果。

查询参数：

- `taskId`

后端读取来源：

- `simulation_single_task`
- `simulation_single_snapshot`
- `simulation_single_confirmed_route`
- `simulation_single_route`
- `simulation_single_route_stop`
- `simulation_single_event_log`

返回原则：

- 尽量复用单店推演原有 `data` 结构，降低前端切换成本。
- A/B 快照以单店审计表为准。
- 两条线路详情以单店线路表、逐店停靠表为准。
- 事件日志以单店事件表为准。

### 10.2 当前前端行为

- 单店时间推演成功后：
  1. 先调用 `POST /simulate/optimize-time`
  2. 再用返回的 `task_id` 调用 `GET /simulate/single-report`
  3. 报告区与弹窗优先展示查库结果
- 若查库失败，前端再回退到当次接口返回值。
