## 1. 文档目的
本文档用于完整说明“经验模块（rengong）”近两天形成的最终成果，覆盖：

- 业务目标与边界
- 数据来源与抽取规则

## 2. 业务目标与边界

### 2.1 目标
从人工调度历史结果（`human_dispatch_routes`）中，提取可复用的“结构经验”，包括：

1. 双配结构（X / X+100）
2. 单配线路结构
3. 模板化统计（结构分布，不输出具体调度解）
4. 经验结果按天落库，支持跨日期批处理与回溯

### 2.2 边界（严格）

- 不改 solver
- 不改 evaluator
- 不改调度主流程
- 不把人工某天方案直接当“调度初始解”
- 只做分析抽取、模板统计、经验展示与经验落库

## 3. 数据来源与核心口径

## 3.1 主数据来源

- 人工调度明细：`human_dispatch_routes`
- 经验页求解对齐明细：`human_dispatch_solver_ready`
- 门店档案补齐：`human_dispatch_solver_profile`（及部分门店基础表）

## 3.2 双配识别口径

按**同一天 + 同车**分组：

- 若存在 `route_id = X` 与 `route_id = X + 100`
- 则定义为一个双配门店组

并计算：

- `first_stores`（X）
- `second_stores`（X+100）
- `core_stores = first ∩ second`
- `extra_stores = second - first`
- `union_stores = first ∪ second`

## 3.3 单配识别口径

按**同一天 + 同车**分组：

1. 先识别并标记全部双配使用的 route_id（X 与 X+100）
2. 剩余 route_id 记为单配线路（single route）

注意：单配不是“无数据”，而是“不参与 X/X+100 对关系”的线路。

## 4. 数据库结构（经验落库三表）

> 以下三表均属于经验模块，不写入调度结果表，不影响主调度。

## 4.1 `rengong_store_groups`（双配结构层）

每条记录表示一个双配门店组。

核心字段：

- `delivery_date`：日期
- `vehicle_id`：车牌/车辆标识
- `base_route_id`：基线路号（较小 route_id）
- `first_route_id`：X
- `second_route_id`：X+100
- `first_store_ids`：一配门店集合（JSON）
- `second_store_ids`：二配门店集合（JSON）
- `core_store_ids`：重合门店集合（JSON）
- `extra_store_ids`：二配新增门店集合（JSON）
- `first_store_count`：一配店数
- `second_store_count`：二配店数
- `core_store_count`：重合店数
- `extra_store_count`：二配新增店数
- `overlap_ratio`：重合率
- `extra_ratio`：二配新增比例
- `created_at`：写入时间

## 4.2 `rengong_single_routes`（单配结构层）

每条记录表示一个单配线路。

核心字段：

- `delivery_date`
- `vehicle_id`
- `route_id`
- `route_name`
- `store_ids`（JSON）
- `store_count`
- `single_type`（`airport / station / metro / other`）
- `created_at`

## 4.3 `rengong_templates`（模板层）

每条记录表示某天的聚合模板（不是具体调度方案）。

核心字段：

- `delivery_date`
- `day_type`（`NORMAL` / `COLD_ONLY`）
- `group_count`
- `avg_first_store_count`
- `avg_second_store_count`
- `avg_extra_ratio`
- `avg_overlap_ratio`
- `single_route_count`
- `airport_count`
- `station_count`
- `metro_count`
- `other_count`
- `single_store_count`
- `small_route_count`
- `medium_route_count`
- `large_route_count`
- `created_at`

### day_type 规则

- 周日：`COLD_ONLY`
- 其余：`NORMAL`

Python 判定口径：`weekday() == 6`（周日）
MySQL 回填口径：`DAYOFWEEK(delivery_date)=1`（周日）

## 5. 后端函数清单（经验模块）

以下函数位于：

- `tools/ga_backend_server.py`

## 5.1 结构抽取函数

1. `_rengong_extract_store_groups(delivery_date)`
   - 输入：日期
   - 输出：
     - `storeGroupStats`
     - `storeGroups[]`
   - 作用：按 X / X+100 提取双配门店组，并生成重合/新增指标

2. `_rengong_extract_single_routes(delivery_date)`
   - 输入：日期
   - 输出：
     - `singleRouteStats`
     - `singleRoutes[]`
   - 作用：提取未参与双配的单配线路

3. `_rengong_build_dispatch_input_simulation(groups)`
   - 输入：双配门店组
   - 输出：
     - `dispatchInputSimStats`
     - `dispatchInputSimulations[]`
   - 作用：构造“调度输入模拟（结构层）”
   - 规则：
     - `wave1_stores = core_stores（一配）`
     - `wave2_stores = core + extra（二配）`
   - 仅用于分析展示，不接调度求解

## 5.2 画像与模板函数

4. `_rengong_build_single_route_portrait(single_routes)`
   - 对单配做画像统计：
     - 分类：机场 / 高铁车站 / 地铁 / 其他
     - 分布：单店、小线(<=3)、中线(4~6)、大线(>=7)

5. `_rengong_build_human_structure_template(...)`
   - 输出三类模板：
     - 双配模板
     - 单配模板
     - 车辆使用模板
   - 仅输出聚合统计，不输出具体车牌绑定方案

## 5.3 落库函数

6. `_ensure_rengong_template_tables()`
   - 作用：确保三张经验表存在；兼容补齐 `day_type` 字段

7. `_rengong_persist_day_templates(delivery_date, store_groups, single_routes, human_structure_template)`
   - 作用：按天“先删后写”落库三张经验表
   - 同时写入 `day_type`

8. `run_rengong_template_batch()`
   - 作用：全量日期批处理
   - 流程：
     1. 读取 `human_dispatch_routes` 全部日期
     2. 逐天提取双配/单配/模板并落库
     3. 逐天做覆盖率校验（human vs experience）
   - 输出：
     - `processed_days`
     - `daily[]`（每天 group/single/missing/extra）
     - `missing_dates[]`

## 5.4 经验接口函数

9. `rengong_day_view(query)`
   - 作用：经验页当天展示主接口
   - 返回包含：
     - 双配、单配、画像、模板、输入模拟、门店双表、车辆表等数据

10. `rengong_summary(query)` / `rengong_list(query)`
   - 作用：经验页摘要与列表支撑

## 8. 关键业务规则与规律沉淀

## 8.1 双配规律

- 双配核心模式：`X / X+100`
- 二配通常是“一配核心 + 少量新增”
- 用 `second_extra_ratio` 与 `jaccard`（重合率）衡量稳定性

## 8.2 单配规律

- 单配是“未配对线路”，不是“空线路”
- 单配可按线路名语义分型（机场/高铁车站/地铁/其他）
- 单配可按店数分布判断任务形态（单店、小线、中线、大线）

## 8.3 日类型规律

- 周日标记为 `COLD_ONLY`
- 非周日标记为 `NORMAL`
- 可直接用于后续分日型建模与策略隔离

## 9. 验收口径与已验证结果

## 9.1 覆盖率验收口径

按天比较：

- `human_dispatch_routes` 当天 `DISTINCT store_id`（人工全集）
- 经验集合：
  - 双配门店组 `store_ids` 并集
  - 单配线路 `store_ids` 并集

检查：

- `missing_count = all_stores - experience_stores`
- `extra_count = experience_stores - all_stores`

## 9.2 已验证状态

全量批处理已可运行，并生成：

- `processed_days`
- 逐日 `group_count/single_count/missing_count/extra_count`

在最新验收中，`missing_dates` 为 0（无缺失日期）。

## 10. 运行与维护建议

1. 每次导入/刷新人工数据后，执行一次全量批处理：
   - `POST /rengong/template-batch`

2. 日常查看某天经验结果：
   - 打开经验页，选日期查询

3. 如发现某天结构异常，优先核查：
   - `human_dispatch_routes` 当天是否缺数据
   - 车号/route_id 是否存在脏值
   - 是否存在异常 route_id 非整数

4. 不建议在经验模块中引入调度求解行为，保持“分析层”职责纯净。

## 12. 术语表

- 双配门店组：同一天同车下，route_id 存在 X 与 X+100 的组
- 单配线路：同一天同车下，不参与 X/X+100 配对的线路
- 核心店（core）：一配与二配共同出现的门店
- 二配新增店（extra）：仅在二配出现、未在一配出现的门店
- 重合率（jaccard）：`|A∩B| / |A∪B|`
- 二配新增比例：`|extra| / |first|`
- 日类型（day_type）：`NORMAL` 或 `COLD_ONLY`

### F. 模块4：店铺收货时间规则（第一张核心可操作表）

表名：店铺收货时间规则
来源：`store_arrival_windows`（接口返回字段）

字段说明：
- 店铺编号（`store_id`）：店铺主键。
- 店铺名称（`store_name`）：可读名称。
- 最早到店（`earliest_arrival_time`）：该店历史最早点。
- 平均到店（`avg_arrival_time`）：按跨午夜口径修正后的均值。
- 最晚到店（`latest_arrival_time`）：该店历史最晚点。
- 允许早到分钟（`early_allowed_minutes`）：相对平均值可提前窗口。
- 允许晚到分钟（`late_allowed_minutes`）：相对平均值可延后窗口。
- 有效样本数（`valid_sample_count`）：参与统计的有效记录量。

#### 可操作控件与行为

1）店铺编号搜索（`storeIdSearch`）
- 输入店号关键字，实时过滤。
- 结果：仅保留编号包含关键字的行。
- 价值：快速定位单店规则。

2）店铺名称搜索（`storeNameSearch`）
- 输入名称关键字，实时过滤。
- 结果：仅保留名称命中的行。
- 价值：不知道编号时按名称查。

3）到店时间起止筛选（`storeTimeStart/storeTimeEnd`）
- 选择时间区间。
- 结果：按平均到店时间过滤。
- 价值：找“晚到店群”或“早到店群”。

4）最小有效样本数（`storeMinValidSamples`）
- 输入阈值。
- 结果：仅保留样本数 >= 阈值的店。
- 价值：避免用小样本店做强判断。

5）点击表头排序（所有列）
- 第一次点击：升序
- 第二次点击：降序
- 第三次点击：恢复默认顺序
- 时间列按夜配跨午夜口径排序，不按字符串。

为什么这样设计：
- 店铺规则是执行落地层，必须支持“快速定位 + 可信度过滤 + 排序比较”。

- 是否脏数据（`is_dirty_candidate`）

#### 可操作控件与行为

1）线路组筛选（`rdGroupFilter`）：all / W1 / W2 / OTHER。
2）线路类型多选（`rdTypeFilter`）：支持 Ctrl/Shift 多选。
3）线路名称联动筛选（`rdRouteNameFilter`）：与线路类型联动，类型变化时重置名称可选项。
4）线路号搜索（`rdRouteIdSearch`）：实时过滤。
5）车辆搜索（`rdVehicleSearch`）：实时过滤。
6）日期范围（`rdDateStart/rdDateEnd`）：限定观察窗口。
7）点击列头排序（`data-rd-key`）：支持日期、类型、线路、店数、里程、装载、布尔状态排序。

为什么这样设计：
- 这张表把“线路结构、里程成本、装载压力”放在同一视角，是运营边界判断主表。

---

### I. 里程与装载表的右键明细能力（关键交互）

交互：在“里程与装载”表任意一行上 **右键**。

触发行为：
- 前端调用只读接口：
  `GET /rengong/route-distance-load-detail?date=...&route_id=...&route_group=...&vehicle_id=...`
- 打开“线路明细”弹窗。

弹窗显示：
1）标题：日期 / 线路号 / 线路名称 / 车辆
2）汇总：店铺数、单程里程、线路装载量、线路类型、是否机场、是否脏数据
3）明细（按 stop_sequence）：顺序、店铺编号、店铺名称、上站到本站km、本站到下站km、原始货量、折算货量、累计装载
4）关闭按钮：关闭弹窗，不影响主表筛选与排序状态

异常行为：
- 若接口失败：弹窗内显示错误提示，不影响主表继续使用。

为什么这样设计：
- 左键用于筛选排序，右键用于下钻明细，兼顾效率与深查。

---

### J. 为什么必须有三张核心表（店铺 / 线路W1 / 里程装载）

1. 店铺收货时间规则：回答“店的时窗边界”。
2. 线路W1装载明细：回答“线的装载边界”。
3. 里程与装载：回答“线在成本与装载上的结构状态”。

三者组合后，可从“店-线-成本”三层做同口径判断。

---

### K. 调度员实操建议（从快到深）

建议顺序：
1. 先选波次（通常W1）。
2. 看装载规则与空间约束，先统一口径。
3. 在线路W1表筛出：超上限 / 机场线 / 脏候选。
4. 在里程与装载表看成本与装载关系。
5. 对可疑线路右键下钻，看店序与分项货量。
6. 回到店铺收货时间规则核对时间边界是否合理。

这样做的结果：先判“是否异常”，再查“异常原因”，最后回到店级边界验证。

### L. 页面边界声明（防误用）

本页不做：
- 自动调度
- 改写调度结果
- 改写主调度规则

本页只做：
- 经验规则展示
- 风险识别
- 线路解释下钻

一句话：
“缘起·性空”是经验边界观察器，不是求解器。

## 附录：登堂页面（dengtang.html）详尽说明（以 2026-04-08 已加载数据为例）

> 目的：让调度同事在不懂模型细节的前提下，也能直接看懂登堂页面“在说什么、怎么点、点完看什么、为什么这样设计”。
> 边界：登堂为只读解释页，不做求解，不写回调度方案。

### A. 登堂页面定位

登堂页不是“再算一遍调度”，而是“还原人工调度当天行为”的解释页。
它回答的问题是：

1. 今天这条线是延续还是调整？
2. 如果调整了，变化发生在哪（店铺集合、顺序、货量、里程）？
3. 这种变化在经验上是否合理，需不需要人工复核？

一句话：
登堂是“解释器”，不是“求解器”。

---

### B. 页面数据来源（调度员应知）

登堂页主要读以下只读链路：

1. `GET /rengong/dengtang-explain?date=YYYY-MM-DD`
   - 返回 `daySummary`（当日汇总）
   - 返回 `routes[]`（每条线路解释）

2. `routes[]` 内部核心结构
   - `reference`：参考日信息
   - `current_snapshot`：今日线路快照
   - `reference_snapshot`：参考日线路快照
   - `diff_summary`：今日 vs 参考日差异摘要

说明：
- 当前第一版参考日口径固定为“前一同类日”（NORMAL 对 NORMAL，COLD_ONLY 对 COLD_ONLY）。

---

### C. 以 2026-04-08 页面为例（你当前界面已加载）

该日摘要（示例）：
- `delivery_date = 2026-04-08`
- `day_type = NORMAL`
- `route_count = 45`
- `exact_copy_count = 21`
- `changed_count = 24`
- `stable_count = 21`
- `risk_count = 0`
- `review_count = 24`

这组数字的含义：
- 当天共 45 条线。
- 21 条属于“稳定复用”（结构未变或高度延续）。
- 24 条发生变化，需要重点解释和复核。
- 当天“需复核量”约等于变化量。

---

### D. 页面每个区域怎么用

#### 1）日期选择区

怎么操作：
- 选择日期，例如 `2026-04-08`。

点击后结果：
- 页面重新拉取当日 `daySummary + routes[]`。

为什么这样设计：
- 解释必须基于“同一天闭环”。

#### 2）KPI摘要区

常见指标：
- 线路数
- 稳定复用
- 发生变化
- 高风险
- 需复核

怎么用：
- 先看当天风险总量，再决定是否需要深查线路明细。

#### 3）筛选区（状态筛选）

常见选项：
- 全部
- 稳定复用
- 发生变化
- 高风险
- 需复核

怎么操作：
- 点一个筛选标签。

点击后结果：
- 仅展示对应类型线路卡片。

为什么这样设计：
- 让调度员先“分层处理”，避免一次看 45 条线造成信息过载。

#### 4）线路卡片区（核心）

每张卡片至少包含：
- 线路号
- 负载状态
- 实际行为
- 建议判断
- 解释文本
- 今日线路快照
- 参考日线路快照
- 差异摘要

怎么用：
- 先看“建议+原因”做快判。
- 再看“今日 vs 参考日”做证据核对。

---

### E. 线路卡片字段逐项解释（重点）

#### 1）基础识别字段
- `route_id`：线路标识。
- `delivery_date`：当前解释日。
- `day_type`：日类型（NORMAL / COLD_ONLY）。

作用：
- 保证解释有明确时空锚点，防止跨天误判。

#### 2）负载与状态字段
- `route_load`：当前线路总负载。
- `load_status`：前端可读状态标签（例如“偏低/正常/偏高”描述）。
- `load_p90`、`load_vs_p90`：用于解释边界位置（若为空则提示画像不足）。

作用：
- 判断“今天是否接近经验上界”。

#### 3）行为判断字段
- `reuse_type`：实际行为类型（如 EXACT_COPY / PARTIAL_CHANGE 等）。
- `decision_hint`：建议动作（例如延续/复核）。
- `review_level`：复核优先级标签。

作用：
- 把结果转成可执行动作，不停留在技术状态。

#### 4）参考日字段（reference）
- `reference_date`
- `reference_type`
- `reference_label`

作用：
- 明确这条解释“在和哪天比”。

#### 5）今日快照（current_snapshot）
- `route_id`
- `route_load`
- `store_count`
- `distance`
- `stores[]`（stop_sequence、store_id、store_name、store_load 等）

作用：
- 还原今天线路真实构成。

#### 6）参考快照（reference_snapshot）
- 同今日快照结构。

作用：
- 给“变化”提供对照基线。

#### 7）差异摘要（diff_summary）
- `added_store_count`：新增店数
- `removed_store_count`：移出店数
- `same_store_count`：相同店数
- `order_changed`：顺序是否变化
- `route_load_delta`：货量变化绝对值
- `route_load_delta_ratio`：货量变化比例
- `distance_delta`：里程变化绝对值
- `distance_delta_ratio`：里程变化比例

作用：
- 把“变了”拆成可复核的具体项，避免模糊判断。

### F. 实操示例（以 2026-04-08、线路 101 为例）

页面可见事实（示例）：
- `route_id = 101`
- `reuse_type = EXACT_COPY`
- `reference_date = 2026-04-07`
- `same_store_count = 6`
- `added_store_count = 0`
- `removed_store_count = 0`
- `order_changed = false`
- `route_load_delta_ratio` 为小幅负变化

调度员怎么解读：
1. 这条线结构未变（店铺和顺序都稳定）。
2. 货量有波动但在线路可承受范围内。
3. 结论是“延续可接受”，无需额外干预。

为什么要这样看：
- 先看结构是否变，再看负载是否异常，优先级更符合现场决策。

登堂当前主交互是：

- 状态筛选
- 线路卡片展开查看

### H. 常见问题与正确用法

1）“看到变化就一定是坏事吗？”
- 不是。变化可能是合理调度行为。
- 重点看变化来源：新增/移出/顺序变化是否与负载变化一致。

2）“为什么有时会提示画像不足？”
- 说明该线路在当前口径下历史画像样本不足或 p90 不可用。
- 此时以结构对比证据为主，不强行做边界判断。

3）“先看什么最省时间？”
- 先看 KPI，再筛“需复核”，最后逐卡片核对差异摘要。

### I. 页面价值总结（给调度培训用）

登堂页的价值不在“替你决策”，而在“把经验变成可解释证据”：

- 从“我感觉这条线不对”
- 变成“我能指出它和前一同类日差在哪、为什么要复核”

它让调度沟通从“经验口头化”变成“证据结构化”。
